# VideoDownloader 第一次审查文档

## 审查范围与验证方式

本次审查范围为 `G:\VideoDownloader` 项目的 Python 源码、配置、打包脚本、Cookie 目录和发布目录结构。重点检查了下载流程、Cookie 管理、Deno/ffmpeg 配置、m3u8 下载、转码流程、构建发布产物和敏感文件。

## 主要问题

### 1. 高风险：仓库内包含真实 Cookie / 登录凭据

项目内存在真实 Cookie 文件：

- `cookies\youtube_cookies.txt`
- `cookies\bilibili_cookies.txt`
- `cookies\douyin_cookies.txt`
- `release\cookies\youtube_cookies.txt`
- `release\cookies\bilibili_cookies.txt`

这些文件包含 YouTube、Bilibili、Douyin 的会话 Cookie，例如 `SESSDATA`、`sessionid`、`__Secure-*` 等。`convert_cookie.py` 中也硬编码了抖音 Cookie JSON。若这些文件被提交、分发或共享，可能导致账号会话泄露。

建议：

- 立即从源码目录和发布目录移除真实 Cookie。
- 将 `cookies/`、`release/cookies/`、`config/task_history.json` 等用户数据加入 `.gitignore`。
- 轮换或退出相关账号登录会话，使已泄露 Cookie 失效。
- 提供 `.example` 示例文件，而不是提交真实 Cookie。

### 2. 功能缺陷：Cookie 自定义路径设置无效

相关文件：`core\cookie_manager.py`

`CookieManager.__init__` 接收 `cookie_dir` 参数，但实际实现中写死为项目根目录下的 `cookies`：

```python
def __init__(self, cookie_dir: str = 'cookies'):
    self.cookie_dir = Path(__file__).parent.parent / 'cookies'
```

这会导致设置页中的 Cookie 存储路径不生效。用户即使修改了配置，解析和下载仍会读取固定目录。

建议：

- 使用传入的 `cookie_dir` 参数。
- 对相对路径按项目根目录或当前工作目录统一解析。
- 设置保存后同步更新 `CookieManager` 实例。

### 3. 功能缺陷：最大并发数配置未生效

相关文件：

- `core\downloader.py`
- `ui\settings_dialog.py`

`Downloader` 保存了 `max_concurrent`，设置页也提供了最大并发数选项，但 `download()` 每次都会直接创建新线程，没有队列、信号量或任务调度限制。

影响：

- 用户设置的最大并发数没有实际效果。
- 连续启动下载会无限创建线程和外部进程。
- 可能造成网络、CPU、磁盘和 UI 响应问题。

建议：

- 使用 `threading.Semaphore(max_concurrent)` 或任务队列限制并发。
- 下载完成、失败、取消后释放并发名额。
- UI 中明确显示排队、下载中、完成、失败等状态。

### 4. 功能缺陷：Deno 自定义路径没有传给 yt-dlp

相关文件：

- `core\downloader.py`
- `core\link_parser.py`
- `ui\settings_dialog.py`

设置页允许用户配置 Deno 路径，但解析和下载时只传入：

```python
--js-runtimes deno
```

没有使用 `self.deno_path`。如果系统 PATH 中没有 `deno`，即使用户在设置里选择了 Deno 可执行文件，YouTube 解析或下载仍可能失败。

建议：

- 明确 yt-dlp 当前版本支持的 JS runtime 参数格式。
- 在解析和下载流程中统一使用配置后的 Deno 路径。
- 当 Deno 不可用时，在 UI 中给出明确错误提示。

### 5. 稳定性问题：m3u8 后台线程直接操作 Qt UI

相关文件：`ui\main_window.py`

m3u8 下载使用普通 `threading.Thread`，完成后直接调用 `_on_download_complete()`：

```python
def download_thread():
    success = self.ffmpeg_handler.download_m3u8(url, output_path)
    self._on_download_complete(task_id, 'completed' if success else 'failed', None)
```

`_on_download_complete()` 会修改按钮、状态栏、表格并弹出消息框。Qt UI 操作应在主线程执行，后台线程直接操作 UI 可能导致随机崩溃、卡死或状态错乱。

建议：

- 将 m3u8 下载改为 `QThread` 或 QObject worker。
- 使用 Qt signal 把完成状态发送回主线程。
- 保持 yt-dlp 下载和 m3u8 下载的线程模型一致。

### 6. 体验/正确性问题：转码进度条按秒数显示

相关文件：

- `core\ffmpeg_handler.py`
- `ui\transcode_dialog.py`

`FFmpegHandler._parse_progress()` 返回的是 ffmpeg 输出中的已处理秒数，不是百分比。UI 直接把该值设置到 0-100 的进度条。

影响：

- 长视频进度值可能超过 100。
- 短视频进度显示不准确。
- 用户无法判断真实完成比例。

建议：

- 转码前获取媒体总时长。
- 将 `current_time / duration * 100` 作为进度百分比。
- 如果无法获取总时长，则显示不确定进度或仅显示状态文本。

### 7. 体验/正确性问题：ffprobe 路径不跟随内置 ffmpeg

相关文件：`core\ffmpeg_handler.py`

`get_media_info()` 固定调用：

```python
cmd = ['ffprobe', ...]
```

但项目只配置了 `ffmpeg_path`，并且默认使用内置 `tools\ffmpeg_bin\ffmpeg.exe`。如果系统 PATH 中没有 `ffprobe`，即使内置 ffmpeg 可用，媒体信息和时长读取也会失败。

建议：

- 根据 `ffmpeg_path` 推导同目录下的 `ffprobe.exe`。
- 允许用户配置自定义 ffprobe 路径，或在内置工具目录中打包 ffprobe。
- 当 ffprobe 不可用时，提供明确错误信息。

### 8. 工程卫生问题：构建产物、发布产物、缓存和二进制混在源码目录

项目目录中存在：

- `build/`
- `release/`
- `__pycache__/`
- `tools/ffmpeg_bin/` 大量二进制文件
- `release\VideoDownloader.exe`
- `release\cookies/` 用户 Cookie

当前没有发现 `.gitignore`。这些内容会导致仓库体积过大、构建结果不可复现、敏感文件容易被误提交。

建议：

- 新增 `.gitignore`，排除构建产物、缓存、用户配置、Cookie、日志和 release 输出。
- 将第三方二进制来源、版本和校验方式写入文档。
- 发布包和源码仓库分离管理。

## 建议修复优先级

1. 立即移除真实 Cookie 和硬编码凭据，补充 `.gitignore`，轮换相关登录会话。
2. 修复 `CookieManager` 使用固定目录的问题，确保设置页 Cookie 路径生效。
3. 实现下载并发限制，让 `max_concurrent` 配置真正生效。
4. 修复 Deno 自定义路径，使 YouTube 解析和下载使用同一套运行时配置。
5. 将 m3u8 下载改为 Qt signal 回主线程更新 UI。
6. 修正转码进度百分比计算。
7. 修正 `ffprobe` 路径解析和错误提示。
8. 清理项目目录结构，区分源码、构建输出、发布包和用户数据。

## 测试建议

后续修复时建议补充以下测试或手动验证场景：

- Cookie 路径改为自定义目录后，解析和下载确实读取新目录。
- 最大并发数设置为 1 时，多个任务按队列顺序执行。
- 系统 PATH 中没有 Deno，但设置了自定义 Deno 路径时，YouTube 解析和下载可用。
- m3u8 下载完成和失败时，UI 不崩溃，状态正确更新。
- 转码进度从 0 到 100 递增，不超过 100。
- 内置 ffmpeg/ffprobe 在无系统 PATH 依赖的环境中可用。
